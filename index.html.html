Case Study — ATM Interface

cBank ATM: Designing for
Every User, Not Just the Average One

A structured look at the design process behind an accessible, error-tolerant public kiosk interface — built as part of the ATM Interface project.

Problem
ATMs fail users at their most stressful moments

Standard ATM UIs are built for the average user — leaving elderly, low-literacy, and first-time digital banking users struggling with cryptic error messages, small text, and no way back when something goes wrong. A single wrong tap can lock a session entirely.

Goal
Make every transaction feel safe, fast, and certain

Design a full ATM interface that works equally well for a 70-year-old first-time user as for a tech-savvy 25-year-old — prioritising legibility, step-by-step confirmation, forgiving error recovery, and multilingual/audio support from screen one.

Research & Insights

What makes ATM flows fail

Pain Point 01
Irreversible actions

No mid-flow cancellation. Users who tap the wrong option have no safe exit — leading to anxiety, session abandonment, and sometimes card lockout.

Pain Point 02
Poor visual hierarchy

Balance figures and entered amounts are visually buried. Error messages appear at the same weight as neutral text — users miss them entirely.

Pain Point 03
Zero accessibility layer

No audio feedback, no language selection on the first screen, no high-contrast option — failing a significant portion of real-world users before the first tap.

Final UI

Key screens

Welcome screen — card insert prompt with language toggle
Main menu — all 5 services visible, large tap zones, audio toggle
Key Features
Design decisions that matter
  • Language + audio toggle on the very first screen, not buried in settings
  • All 5 core actions on one menu — zero hidden sub-menus
  • Personalised greeting with masked account number builds security confidence
  • Icon-led buttons with consistent minimum tap zones (80×80px+)
  • Step indicator on every screen so users always know where they are
HCI Heuristics Applied
Principles in practice
  • Visibility of system status — every action confirms immediately
  • User control and freedom — cancel is always one tap away
  • Error prevention over error correction — confirmations before irreversible steps
  • Recognition over recall — icons paired with labels throughout
  • Aesthetic and minimal design — dark background reduces outdoor glare
↓ 40%
Estimated task completion time

Outcome

By surfacing all actions upfront, removing hidden sub-menus, and adding language and audio support from the first screen, the redesigned interface significantly reduces cognitive load across all user types. Elderly and low-literacy users can complete a withdrawal or balance check without assistance — which was the primary success metric for this project.

What I learnt from this project

  • Accessibility cannot be an afterthought — it must start at the wireframe stage
  • Error prevention is always more effective than writing better error messages
  • Testing with edge-case users reveals flaws that are invisible to tech-savvy testers
  • Dark UI on public kiosks reduces glare but demands careful contrast management
  • Every extra screen in a flow is another chance for a user to drop off
  • Designing for the least-confident user makes the interface better for everyone
